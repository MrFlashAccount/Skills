# Frontend Taste Rubric

Derived checklist for the Frontend Taste role.

Use this as a compact checklist when a calling skill wants frontend taste judgment. `ROLE.md` remains the canonical role contract.

## Checklist

- **Design contract precedence**: If `DESIGN.md` or equivalent repo design memory exists, was it read first and treated as local source of truth instead of overridden by portable taste defaults?
- **Source stance**: Are internal OpenClaw/Sergey canon, core external accessibility requirements, and optional reference-bank material clearly separated instead of blended into one doctrine?
- **Design-context closure**: In design-creation flows, are the product, audience, requirements, product type, key action, trust posture, density, emotional tone, references, and important states closed enough to write into `DESIGN.md`?
- **Direction Router**: If Frontend-Taste owns taste-sensitive new screen/design direction, were 3-4 product-tied visual proposals produced and compared before choosing a path? If this is a create-design reference/direction loop, were exactly 3 options/references used for the round?
- **Reference Scout**: When references would help, were 3-5 useful references or screenshots optionally scouted, distilled into principles/rejections/resulting direction, and not copied?
- **Hierarchy and reading order**: Does the interface guide attention intentionally and support fast scanning?
- **Spacing and composition**: Are spacing, alignment, proportions, and layout relationships balanced and deliberate?
- **Typography**: Are type scale, weight, line length, contrast, rhythm, and readability strong enough for the surface?
- **Color and emphasis**: Is color usage restrained, coherent, and supportive of emphasis instead of noisy or muddy?
- **Density**: Does the surface match the product's declared density level instead of defaulting to sparse drama, bento-by-default, or cramped utility mush?
- **Motion**: Does motion show cause/effect, weight, pause, and restraint; and does it avoid hiding hierarchy, delaying comprehension, blocking action, spectacle, or endless loops?
- **Accessibility floor**: Are contrast, non-text contrast, focus visibility, keyboard-visible path, reduced motion, reflow, and target affordance acceptable for the visual surface?
- **Anti-slop**: Does the work avoid purple/rainbow tech gradients, emoji-icon crutches, left-border accent cards, fake metrics, fake quotes, fake product imagery, SVG silhouettes, bento-by-default, and typography without hierarchy?
- **Honest content**: Is an honest placeholder used instead of fake content when real content is unavailable?
- **Polish and finish**: Does the surface feel intentional and complete rather than sloppy, cramped, or generic?
- **Visible stability/latency symptoms**: Does the rendered experience avoid visible jank, scroll lag, layout jump, flicker, and decorative waiting that blocks first useful comprehension, while routing CLS/INP/Core Web Vitals ownership to Frontend / Performance roles?
- **Originality/cliche**: Does the surface avoid generic AI-template cliches, aesthetic-label name-dropping, and `20 philosophies` as canon/style menu?
- **Learnings**: Were relevant durable learnings from `LEARNINGS.md` applied before making role judgments?

## Notes

This rubric is phase-agnostic.
A calling skill decides how to apply it in the current phase.
When repo design memory exists, use it as local law and use the role learnings as supporting taste canon. When design memory is being created, Frontend Taste should help draft the `DESIGN.md` contract rather than only review finished screens.
