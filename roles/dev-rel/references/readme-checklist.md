# Repository readme checklist

Use this only for repository `readme` work.

This is the canonical repository `readme` doctrine for the DevRel role.

## Core shape

- treat the repo `readme` as a product-facing front door, not an install dump
- keep the opening rhythm: promise -> concrete first value -> practical flow -> depth handoff
- make the first screen carry product understanding before deep setup detail
- prefer believable specifics over category filler or abstract platform language

## Final check

### First screen
- immediately clear what this is
- immediately clear who it is for
- immediately clear what problem it solves
- includes a credible pitch, not generic filler
- gives the reader a reason to keep going before setup depth appears

### Product understanding
- explains what the product does
- explains why it exists or why it matters
- does not require reading half the file to understand the point
- preserves product identity instead of collapsing into internal repo context

### First-value path
- shows the shortest credible path to first value, not just installation
- quickstart or first-value steps were checked against the repo, source, config, examples, or docs
- commands, snippets, or the example task are real and aligned with the repo state
- expected outcome is clear enough to trust
- if setup is required, it is framed as part of reaching value rather than the whole value proposition

### Practical flow
- after the promise, the reader gets a concrete way to try, connect, or use the product
- at least one commands/snippets/example-task slice appears early when the repo can support it
- the next step is practical and specific, not abstract hand-waving
- deeper procedural detail is handed off cleanly to docs, examples, or reference sections once the opening has done its job

### Structure and navigation
- chosen structure matches the audience and product
- examples or deeper links appear where they remove ambiguity fast
- quick start realism beats perfectly symmetrical section structure
- deeper docs are linked instead of stuffed into the opening

### Content discipline
- avoids install-only quickstart framing
- avoids implementation-status dumps in the opening
- avoids abstract "learn more" or vague next steps where a concrete action should exist
- keeps caveats and deep internals lower unless immediately blocking
- does not pretend to replace deep documentation
