# Launch handoffs

A launch plan is incomplete if downstream owners cannot execute it.

## DevRel handoff

Route to `devrel-copywriter` when the launch needs:
- repository `readme` framing changes
- developer-facing announcement copy
- technical audience message hierarchy
- proof-sensitive product framing for code-first readers

Provide:
- audience and launch goal
- approved claim boundaries
- proof sources and screenshots
- what should be clear on first screen

## Docs handoff

Route to `docs-writer` when the launch depends on:
- onboarding updates
- setup or migration instructions
- feature tutorials, quickstarts, or reference updates
- activation paths that must work on day one

Provide:
- the launch promise
- the first user success the docs must unlock
- prerequisite changes and source of truth
- known risk points or support traps

## Sales handoff

Route to `sales-enablement` when the launch affects:
- outbound and follow-up collateral
- demo flow changes
- pricing explanation and packaging talk tracks
- objection handling for new availability or new limits

Provide:
- audience segments
- what changed and why it matters
- approved proof and tradeoffs
- pricing/packaging notes if relevant

## Customer-research handoff

Open a customer-research track before or after launch when you still need to validate:
- which segment feels the pain most
- willingness to pay or package fit
- objection patterns and lost-deal reasons
- whether the promise matches actual user language

Capture:
- hypotheses to test
- target respondent type
- exact questions or unknowns
- decisions blocked by missing evidence

If the repo needs a reusable research packet first, use `research-critic` to structure the open questions and evidence gaps.
