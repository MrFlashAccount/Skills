# Strategic DDD

Use Strategic DDD when the hard part is domain boundaries, ownership, and language, not just code layering.

Its main outputs are bounded contexts, ubiquitous language, context relationships, and an ownership model that keeps different meanings from bleeding together.

## Use it when

- multiple contexts or teams use overlapping words differently
- context boundaries and translation matter more than framework layering
- a context map or explicit ownership split would change structural decisions

## Keep it light when

- one language and one owner still fit the whole slice
- the split would be theater with no real boundary pressure

## Sources
