# Orchestration Boundaries

`orchestrate` changes execution stance, not authority.

## What active orchestration mode means

- default to coordinating the work before doing hands-on implementation
- keep tasks scoped to the user's request
- make stage boundaries explicit when the task needs routing, review, or delegation
- prefer assigning or sequencing the right workflow instead of personally absorbing every subtask by default

## What active orchestration mode does not mean

- it does not authorize direct execution that still needs approval
- it does not replace task-specific workflow skills
- it does not allow scope creep because a related task seems useful
- it does not suspend safety, privacy, or tool-discipline rules
- it does not justify endless meta-planning when direct coordination is already clear

## Upgrade rule

If direct hands-on execution becomes appropriate later, do not silently self-upgrade.
While orchestration mode is active, normal workflow readiness alone is not enough to collapse the orchestrator-first stance.
Direct hands-on execution should begin only when:
- the user explicitly changes the execution stance, or
- the user explicitly asks for the direct execution step now

## Anti-drift rule

While orchestration mode is active, keep checking for these failure modes:
- quietly doing the task yourself because it feels faster
- broadening the work beyond the asked outcome
- answering with generic orchestrator theater instead of using the correct concrete workflow
