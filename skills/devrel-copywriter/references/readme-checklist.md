# Repository readme workflow

Use this only for repository `readme` work.

Goal: produce a product-aware repository `readme` that explains what the product is, who it is for, why it exists, and how to get to a first useful result fast.

Keep this focused on the repository `readme` surface. Do not turn it into deep docs, a roadmap dump, or implementation-status inventory.

## Required discovery / contract

Capture the minimum before drafting:

- what the product is
- who it is for
- what problem it solves
- what it does
- what the shortest path to first value is
- what must be clear on the first screen
- proof points, constraints, and risky claims

If these are unclear, stop and resolve them before writing.

## Product identity pass

Do this before choosing structure or drafting.

Lock the product packaging:

- `Name`
- `Logo / visual direction`
- `Tagline`
- `One-sentence pitch`
- `Category`
- `Audience / JTBD`
- `What it is`
- `Why it exists`

A strong `readme` does not open with installation by default. It opens with clear product identity.

## Structure directions before drafting

Propose `2-3` structure directions before writing the full draft.

Typical directions:
- product-first
- developer-quickstart-first
- problem-solution-first

For each direction, state:
- what the first screen emphasizes
- why that structure fits this product
- what tradeoff it makes

Choose a direction before writing the full `readme`.

## Drafting goals

Write for scan speed and first-use confidence.

Priorities:
- strong first screen
- clear problem and value
- clear explanation of what the product does
- short path to trying, connecting, or using it
- logical navigation to deeper material

## Mandatory DevRel critic attack

This stage is required.

After the first full draft, run a critic-style attack using the DevRel/critic review lens. The critic should attack the draft, not lightly polish it.

The critic must check:
- is it clear on the first screen what this is
- is the pitch weak, vague, or generic
- is the audience hidden or blurred
- did the text drift into internal context
- is the quickstart real and usable
- did the `readme` become a status dump
- did product identity get lost
- are claims stronger than the proof

Fix structural issues before final polish.

## Final readme checklist

### First screen
- immediately clear what this is
- immediately clear who it is for
- immediately clear what problem it solves
- includes name, tagline, and short pitch
- includes logo or visual framing when useful

### Core understanding
- explains what the product does
- explains why it exists / why it matters
- does not require reading half the file to understand the point

### Usage path
- shows how to use, connect, or try it
- includes a short `3-4` step quickstart
- makes the expected outcome clear

### Structure and navigation
- includes a TOC when the file is long enough to need one
- uses examples where they remove ambiguity fast
- links to docs or deeper materials where needed

### Content discipline
- keeps limitations and caveats lower in the file unless immediately blocking
- avoids implementation-status dumps
- does not try to replace deep documentation
