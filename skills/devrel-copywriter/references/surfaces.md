# Surface heuristics

Apply only the parts that match the task.

## README intro

- Explain what it is in one plain sentence.
- State why it exists before listing features.
- Keep the first screen scannable.
- Push advanced internals down unless they are the reason to adopt.
- Make the quick start feel possible.

## X / Twitter post or thread

- Open with the strongest concrete idea, not throat clearing.
- Respect the reader's attention; every line must earn its place.
- One post = one clear angle.
- Threads need momentum; each post should pull into the next.
- Avoid obvious founder cringe, fake humbleness, and empty build-in-public theater.

## Changelog / release note

- Lead with what changed for the user.
- Keep chronology below impact.
- Mention migration or breakage early if relevant.
- Do not bury caveats after the excitement.

## Landing section

- Start with outcome, not implementation.
- Keep subhead useful, not decorative.
- Feature bullets should map to a reader concern.
- Remove generic words like seamless, powerful, robust unless they are backed by something concrete.
